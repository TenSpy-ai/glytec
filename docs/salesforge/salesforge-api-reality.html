The Contact Data Model
Only 8 fields exist. Everything else goes in customVars.
Both POST /contacts and PUT /import-lead accept extra fields like jobTitle and phone — they return 201/204 — but the data silently vanishes. It's not on the model.

API Gotchas That Will Bite You
Rules an AI builder needs to know
Filters don't filter
Every list endpoint (status on sequences, tags on contacts, reply_type on threads, status on mailboxes) accepts filter params but returns the full unfiltered dataset. You must paginate everything and filter client-side.
PUT, not POST on sub-resources
POST /sequences/{id}/mailboxes → 404. POST /sequences/{id}/contacts → 404. Must use PUT for mailbox assignment and contact enrollment. POST only works for top-level creates.
PUT /steps replaces everything
Sending steps array deletes any steps you omit. Always send the full array. Same applies to PUT /mailboxes — empty array [] clears all assignments.
Changing productId crashes the API
PUT /sequences/{id} with productId returns 500 Internal Server Error. Language and timezone are silently ignored on PUT. These are set-once at creation.
language is not "en"
Sequence creation requires language: "american_english" — not "en", not "english". Full list: american_english, british_english, dutch, italian, czech, hungarian, swedish, ukrainian, german, danish, romanian, french, russian, estonian, finnish, latvian, japanese, brazilian_portugese, norwegian, spanish, polish, lithuanian.
linkedinUrl: omit, don't empty-string
On import-lead: omitting the field or passing null works fine. Passing "" (empty string) → 400 validation error. This applies to import-lead only.
Variant fields are secretly required
PUT /steps requires distributionStrategy (equal|custom), label, and status (active|paused) on every variant. Swagger doesn't mark them required. Missing any → 400.
Can't activate without mailboxes
PUT /status → active fails with "sequence without mailboxes" if no mailboxes are assigned. Also: can't go back to draft, can't manually set completed.
Duplicate contacts are upserts
POST /contacts with an existing email returns 201 with the same lead ID. Updates name/company on the existing record. Not an error — it's upsert behavior. But duplicate enrollment increments leadCount (same contact counted twice).
Bulk create requires tags
POST /contacts/bulk requires a tags array on every contact or you get 400. Single POST /contacts doesn't require tags (auto-adds pa-YYYYMMDD).
No Bearer prefix
Auth header is Authorization: <raw key>. Adding Bearer prefix will fail. Every endpoint is workspace-scoped: /workspaces/{id}/...
customVars is your Swiss Army knife
Any key-value pairs in customVars persist and return on every GET. Use this for: job title, phone, persona frame, EHR vendor, signal, bed count, facility name. It's the only extensible field on the contact model.
Still Manual in V1
No API path exists for these
LinkedIn outreach (Day 5 + Day 10)
Zero LinkedIn endpoints in the Salesforge API. Connection requests and follow-up messages are sent by Clayton/Reagan manually from their profiles. This is 40% of the outreach arc.
Reply routing to AEs
Webhooks fire for positive_reply and email_replied, but routing an interested reply to an AE within 4 hours requires your own alerting system (Slack webhook, email, etc.).
Salesforce writeback
No native Salesforge → Salesforce sync. Opens, replies, bounces, and sequence activity must be pushed to Salesforce by your own code catching Salesforge webhooks and writing to the SF API.
Mailbox reconnection
Several mailboxes show disconnectReason errors. The API can detect disconnection but reconnecting requires OAuth flows through the UI or Warmforge.
Bottom Line
The 3-email sequence is fully automatable. LinkedIn is 100% manual.
The kill-switch, webhook routing, DNC sync, and SF writeback all work — but your code is the orchestrator.
Salesforge is the sending engine. Everything else is you.
