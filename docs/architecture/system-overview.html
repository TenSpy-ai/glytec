How the washing machine works (6 steps)
The problem: Multiple sources return the same person with different data. Without explicit match logic, you either spam someone twice or lose a valid contact.

The fix: Clay enforces a 6-step cleaning process on every record:
1. Dedup & Merge
Match keys in priority: email → LinkedIn URL → name + Definitive ID → name + domain. Merge rules: ZoomInfo email wins, most specific title wins, DH Definitive ID for facility assignment. One golden record per person.
2. Email Validation
NeverBounce / ZeroBounce waterfall. Invalid emails never enter a sequence. If bounce exceeds 5%, all sends auto-pause. Zero tolerance.
3. Suppression Check
Checked against: current customers, active opportunities, competitors, prior opt-outs, and do-not-contact flags. Suppressed contacts are flagged (not deleted — audit trail matters) but never enter a sequence.
4. LinkedIn Cross-Reference
Every contact is checked against their LinkedIn profile. If they've changed titles or moved to a different company, the contact is flagged and marked stale. Prevents sending to people who no longer hold the role we're targeting.
5. ICP Title Match
Fuzzy-match every contact against 66 ICP titles. Assigns category (Decision Maker vs Influencer), function type, and job level. Non-matching titles flagged for review.
6. Account Mapping
Links contacts to the correct Salesforce account via Definitive ID and parent/child hierarchy. Handles IDN structures where one contact may sit at a specific facility within a larger health system.

V1 sequence structure
API over UI — the UI is unreliable. Every operation is programmatic.
  • Day 0: Email 1 — personalized cold email with signal-based hook
  • Day 3: Email 2 — different angle, same pain, new proof point
  • Day 5: LinkedIn connection request (if URL exists)
  • Day 7: Email 3 — direct CTA + case study
  • Day 10: LinkedIn follow-up (if connection accepted)
Kill switch: Bounce >5% or spam >0.3% in any 24-hour window auto-pauses everything. Monitored via Salesforge webhooks + Warmforge placement tests. Auto-pause logic runs on Datagen.dev.
Salesforge + Warmforge APIs (how it runs programmatically)
Two APIs, one platform. Called via Claude Code scripts or Datagen SDK — never through the UI.
Salesforge API — Outbound
Contacts: POST /contacts/bulk or PUT /sequences/{id}/import-lead (creates + enrolls in one call). Contact model has 8 fields: id, name, email, company, linkedinUrl, tags, customVars. jobTitle does NOT persist — use customVars.
Sequences: POST /sequences to create (requires productId, language: "american_english", timezone). PUT /sequences/{id}/steps for email content. PUT /sequences/{id}/mailboxes to assign warmed inboxes (must be PUT, not POST).
DNC sync: POST /dnc/bulk pushes Clay suppression list — body is {"dncs": ["email"]} (array of strings). Enforced at sequencer level.
Metrics: GET /sequences/{id}/analytics (requires from_date + to_date) + GET /sequence-metrics (workspace-level) for sent/opened/replied/bounced.
Webhooks: POST /integrations/webhooks captures opens, replies, bounces in real-time → feeds Datagen auto-pause + Salesforce activity. 10 event types: email_sent, email_opened, link_clicked, email_replied, linkedin_replied, contact_unsubscribed, email_bounced, positive_reply, negative_reply, label_changed.
Warmforge API — Deliverability
Mailbox onboarding: POST /mailboxes/connect-oauth2 for Gmail/Outlook (Clayton + Reagan inboxes).
Warmup monitoring: GET /mailboxes/{address}/warmup/stats — daily sent/reply/spam by provider. Must hit “warm” status before any sequence.
Placement tests: POST /placement-tests sends to seed addresses. Returns inbox vs spam counts by provider. Run before every campaign launch.
Reputation: GET /mailboxes/{address} returns DNS health + reputation scores. Drop = mailbox pulled from sequences.
