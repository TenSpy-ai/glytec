What the HospitalView API delivers & how it maps to the infrastructure
The Commercial API exposes HospitalView data via GET and POST endpoints, all keyed by Definitive ID. Every capability below maps directly to a piece of the GTM infrastructure.
  • Enterprise Account Segmentation — staffedBeds, hospitalType (Short Term Acute Care, etc.), ownership (Voluntary Nonprofit, Investor-Owned), academicMedicalCenter, traumaCenter level, firmType. → Powers segment assignment (Enterprise/Strategic vs Commercial), bed-weighted scoring (20% of account score), and facility-level targeting.
  • Network / IDN Hierarchy — networkId, networkName, networkParentId, networkParentName, networkOwnership, affiliated facilities. → Powers parent/child account mapping. One API call returns every facility under a health system — critical for IDN-level selling where a champion at one hospital opens doors across the network.
  • Technology Install (EHR) — vendor, product, vendorStatus (Installed Vendor), implementationYear, technology category. → Powers EHR-segmented messaging. Epic shops get the “DIY calculator liability” angle. Cerner shops get the “Oracle transition risk” angle. MEDITECH shops get the “modernization” angle. This is Layer 2 of the 3-layer message composition.
  • Executive Contacts — standardizedTitle, positionLevel (C-Level, VP, Director), department, primaryEmail, directPhone, linkedinProfileURL, executiveId. → Feeds the contact enrichment pipeline. DH provides facility-level leadership mapped by Definitive ID. Email quality is poor (ZoomInfo overwrites), but titles, departments, and LinkedIn URLs are high-value for ICP matching.
  • Quality & Hospital Compare Signals — readmissionPenalty, VBP Adjustment, HAC Rate Penalty, CMS star ratings. These are the Hospital Compare-related objects exposed in the API. → Powers CMS quality scoring (15% of account score) and the “CMS is publicly reporting your glycemic harm outcomes” messaging angle. Hospitals with active penalties are higher-urgency targets.
  • Financials — staffedBeds, numEmployees, number of discharges, operating margin. → Powers the financial health matrix. Unprofitable facilities (55.9% of universe) get ROI-led messaging ($20K/bed savings). Profitable facilities get CMS quality-led messaging. This is Layer 3 of the 3-layer message composition.
  • News & Intelligence — intelligenceType (M&A, Affiliations/Partnerships, Clinical Trials/Research, Leadership Changes), publicationDate, newsEventTitle, bodyOfArticle. → Powers trigger-based outbound timing. An EHR migration announcement = receptive moment for the “don't build a DIY calculator” message. A new CNO = new evaluation cycle = buying window. These signals feed into Clay to trigger personalized sequences at the moment of maximum receptivity.
  • RFPs & Certificates of Need — type (Request for Proposal), category (Technology Services, Real Estate, Construction), classification (Solicitation, Certificate of Need), datePosted, dateDue, contact info. → Another trigger signal source. Technology RFPs and new-build Certificates of Need signal active buying cycles. Construction/renovation = new units = new deployment opportunities.
  • Memberships — GPO, ACO affiliations by type and subtype. → Useful for network-level targeting and understanding purchasing dynamics (GPO contracts may influence vendor selection).
API supports both GET and POST. Properties are queryable — we can filter by hospitalType, state, vendor, networkParentId, etc. This means we can programmatically pull just the accounts that match our target criteria rather than processing the entire universe.
Web Scraping
Firecrawl hits hospital leadership pages. Clinical titles live here — CDCES, nurse educators, clinical pharmacists. ZoomInfo can't see them. Perplexity fills signal gaps (press releases, strategic announcements).
How Firecrawl & Perplexity fill signal gaps (with examples)
Firecrawl — structured scraping of hospital websites
  • Leadership directories: Scrapes /about/leadership or /our-team pages to pull names, titles, and credentials for clinical roles (Diabetes Educator, Clinical Pharmacy Director, Nurse Manager — Endocrine Unit) that don't exist in ZoomInfo or DH.
  • Service line pages: Hits /services/endocrinology or /programs/diabetes-management to confirm whether a facility has a dedicated glycemic management program — a key receptivity signal.
  • News/press rooms: Scrapes /newsroom or /press-releases for EHR migration announcements, new unit openings, or executive hires that DH News hasn't picked up yet.
  • Job postings: Scrapes career pages for open endocrinology or clinical informatics roles — a facility hiring for these positions signals an active investment in glycemic care.
Perplexity — AI-powered research for unstructured signals
Note: Claygent can substitute for Perplexity in all of these use cases.
  • "Has [Hospital] announced an EHR migration in the last 6 months?" — Synthesizes across press releases, trade publications, and news sites that Firecrawl would need multiple targeted URLs to cover.
  • "Who is the Chief Nursing Officer at [Hospital]?" — Finds the answer even when the hospital website doesn't have a clean leadership page, by cross-referencing LinkedIn, conference speaker bios, and press quotes.
  • "Does [Health System] have a glycemic management committee or insulin safety initiative?" — Surfaces published quality reports, patient safety committee mentions, and conference presentations that indicate existing awareness of glycemic management.
  • "What recent CMS penalties or quality concerns affect [Hospital]?" — Aggregates CMS data, news coverage, and patient safety reports into a single answer — useful when the DH quality data lags behind real-time events.
  • "Has [Health System] made any strategic announcements about diabetes or endocrinology programs?" — Catches expansion plans, partnership announcements, or grant-funded initiatives that signal budget allocation toward glycemic care.
When to use which: Firecrawl when you know the URL and want structured extraction. Perplexity (or Claygent) when the signal could live anywhere and you need AI to find and synthesize it.

What triggers the push from Clay to Salesforce
Two workflows push data:
  • Account Enrichment & Scoring (Workflow 1): Triggered when a new account is added to the target list or on monthly re-score cycle. Pulls DH data, computes score, checks suppression, pushes to SF.
  • Contact Enrichment Waterfall (Workflow 2): Triggered when an account scores Priority or Active AND contact coverage is below 15. Runs the full waterfall + wash, pushes clean contacts to SF.

Phase 5 — Contacts Enter Salesforge
Sequencing Engine
Salesforge
Priority and Active contacts push from Salesforce into Salesforge sequences. API over UI — everything is programmatic.
How the Salesforge + Warmforge APIs work (programmatic operations)
Two APIs, one platform. Both called via Claude Code scripts or Datagen SDK — never through the UI. Salesforge API — Outbound:
  • Push contacts into sequences — PUT /sequences/{id}/import-lead creates + enrolls in one call (preferred), or POST /contacts/bulk then PUT /sequences/{id}/contacts to assign by ID. Contact model has 8 fields only — jobTitle doesn’t persist, use customVars instead.
  • Create & manage sequences — POST /sequences to create (requires productId, language: "american_english", timezone). PUT /sequences/{id} only updates name and tracking flags — ignores steps/mailboxes. PUT /sequences/{id}/steps is a separate endpoint for email content (replaces all steps).
  • Assign warmed mailboxes — PUT /sequences/{id}/mailboxes assigns Clayton’s or Reagan’s warmed inbox to a sequence. Must be PUT, not POST (POST → 404). Only mailboxes with “warm” status from Warmforge get assigned.
  • Sync DNC lists — POST /dnc/bulk pushes suppressed contacts from Clay. Body: {"dncs": ["email"]} — array of strings, not objects.
  • Pull sequence metrics — GET /sequences/{id}/analytics (requires from_date + to_date query params) and GET /sequence-metrics (workspace-level aggregates) return sent/opened/replied/bounced counts. Feeds the reporting layer.
  • Capture events via webhooks — POST /integrations/webhooks (NOT /webhooks) registers listeners for 10 event types: email_sent, email_opened, link_clicked, email_replied, linkedin_replied, contact_unsubscribed, email_bounced, positive_reply, negative_reply, label_changed. Events flow back to Datagen for auto-pause logic and to Salesforce for activity tracking.
  • Handle replies — GET /threads lists conversation threads with replyType (ooo, lead_replied). Programmatic reply not supported via API — replies are manual in V1.
Warmforge API — Deliverability:
  • Connect inboxes — POST /mailboxes/connect-oauth2 for Gmail/Outlook. Clayton and Reagan’s dedicated inboxes onboard here.
  • Monitor warmup progress — GET /mailboxes/{address}/warmup/stats returns daily sent/reply/spam counts by provider. Inbox must hit “warm” status before entering any sequence.
  • Run placement tests — POST /placement-tests sends test emails to seed addresses. GET /placement-tests/{id} returns inbox vs spam counts by provider (Gmail, Outlook, etc.). Run before every campaign launch.
  • Track reputation — GET /mailboxes/{address} returns DNS health data and reputation scores. If reputation drops, mailbox is pulled from sequences automatically.
Execution path: Claude Code scripts call these APIs directly (REST + API key), or Datagen SDK wraps them as tool executions. Either way, zero manual UI interaction.

The feedback loop
This isn't a one-way pipeline. Response data flows back and refines everything upstream:
  • Open/reply rates by segment → refine which EHR × pain × persona combos work best
  • Bounce data → feeds back to Clay suppression list (auto-flag invalid emails)
  • Meeting conversion by signal type → refine which DH triggers predict receptivity
  • Opt-outs → immediate sync to Salesforce → Clay suppression → contact never re-enters
Pat gets the dashboard. Clayton gets the feedback loop. Same data, different views.
